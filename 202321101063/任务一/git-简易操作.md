# Git 简易操作教程

> 本文是我自学 Git 时整理的笔记，内容只覆盖日常最常用的操作，适合 2~3 人小团队使用。

---

## 一、Git 是什么

Git 是一个**版本管理工具**。你可以把它理解成一个"有记忆的存档系统"：

- 每次修改完代码，存一个档（叫 commit）
- 不满意可以回到任意一个旧存档
- 多人各自存档，最后合并到一起，不会互相覆盖

---

## 二、三个基本概念

| 概念 | 简单理解 |
|------|----------|
| **工作区** | 你电脑上的项目文件夹，改代码的地方 |
| **暂存区** | 一个"购物车"，决定哪些修改要放进下一个存档 |
| **仓库** | 存档的集合，记录了你所有的 commit |

一句话流程：**改文件 → 加入暂存区 → 提交存档**

---

## 三、日常必用命令

### 3.1 下载仓库（只需一次）

```bash
git clone <仓库地址>
```

示例：
```bash
git clone https://gitee.com/qsbian/DYZG.git
```

### 3.2 查看当前状态

```bash
git status
```

这是最常用的命令，告诉你：
- 哪些文件被修改了
- 哪些文件还没加入暂存区
- 工作区是否干净

### 3.3 提交修改（三步走）

```bash
# 第一步：加入暂存区
git add 文件名
git add .          # 添加所有修改

# 第二步：提交存档
git commit -m "写清楚这次改了什么"

# 第三步：推送到云端
git push origin main
```

### 3.4 同步别人的更新

```bash
git pull origin main
```

在开始工作前先拉取一次，避免冲突。

## 四、2~3 人团队协作场景

### 场景一：日常开发（最频繁）

> 假设你和队友 A、队友 B 三个人一起开发。

**每个人每天的标准流程：**

```bash
# 1. 开工前，先拉取最新代码
git pull origin main

# 2. 改自己的代码...

# 3. 提交
git add .
git commit -m "修复了登录页面的bug"

# 4. 推送
git push origin main
```

### 场景二：避免冲突的小技巧

小团队建议 **各人改各人的文件**，减少同时改同一个文件的概率。

```
项目目录/
  ├── 张三/          ← 张三只改这个目录下的文件
  │    └── 功能A/
  ├── 李四/          ← 李四只改这个目录下的文件
  │    └── 功能B/
  └── 王五/          ← 王五只改这个目录下的文件
       └── 功能C/
```

### 场景三：推送前发现别人已经更新了

```bash
# 先拉取
git pull origin main

# 如果有冲突，解决冲突后
git add .
git commit -m "合并冲突"
git push origin main
```

### 场景四：改坏了想回退

```bash
# 回退某个文件到最后一次 commit 的状态
git checkout -- 文件名

# 查看所有历史提交
git log --oneline

# 回到某个历史版本（只查看）
git checkout <commit编号>
```

## 五、常见问题速查

| 问题 | 解决方法 |
|------|----------|
| `git push` 失败，提示需要先 pull | `git pull origin main` 后再 push |
| 忘记加了哪些文件 | `git status` 看一眼 |
| commit 信息写错了 | `git commit --amend -m "新的信息"` |
| 想取消已 add 的文件 | `git reset 文件名` |
| 不知道仓库地址 | `git remote -v` 查看 |

## 六、日常工作口诀

```
开工先 pull，收工必 push
改前看 status，改后写清楚 commit
一人一目录，冲突自然少
```
