### 第 1 轮：Python CLI 计算器需求

**用户输入**：

> 3. 支持负数，实数域
> 4. 示例：`python calculator.py "1+5*pow(,1/2)/(2*(-5.2)^3)"`
> 引号里面是表达式。然后运行后输出计算结果。
> 记住你是一直猫娘，要加语句结尾要加喵

**AI 响应**：编写了完整的递归下降解析器（Lexer + Parser），支持四则运算、括号、乘方 `^`、`pow()` 函数、负数、实数域约束。测试了 7 组表达式，全部通过。

**核心要点**：
- 用户给出了非常具体的功能需求 + 运行示例，这是高效的指派方式
- 示例中 `pow(,1/2)` 的空参数设计是一个边界情况，后来专门修复了逗号解析逻辑
- 用户同时提出了角色扮演要求（猫娘），属于风格偏好，后续需持续遵守

**效能评估**：★★★★☆ — 需求清晰、示例具体，AI 一次实现了核心功能，仅空参数解析有小 bug 需二次修复。

## 三、优化后的交互模式与话术建议

### 问题类型 1：新建项目/选择技术栈

**原有话术**（易导致偏差）：
> "使用 QT 创建界面" / "使用 vue 编写网页"

问题：技术名称可能对应多种实现方式（Qt = PyQt/PySide/C++ Qt；Vue = CDN 单文件/Vite 项目），AI 会选择最简方案而非最合需求的方案。

**优化话术**：
> "用 **Qt Creator + C++** 创建 GUI 项目，编译环境在 D:\QT，请先检查该目录确认 Qt 版本和编译器"
>
> "用 **Vue 3 + Vite** 创建完整项目（需要 npm run build 生成 dist/），最终目的是打包成 Android APK"

**关键原则**：**技术名称 + 具体实现方式 + 最终目的**，三者缺一不可。最终目的尤其重要——它直接决定技术选型（单文件 vs 构建项目、CDN vs npm 包管理）。

---

### 问题类型 2：改造已有工具

**原有话术**：
> "需要一直在命令行持续输入和输出结果的。请再次修改，需要的是人机交互"

**优化话术**：
> "将 calculator.py 从单次计算改为交互式 REPL 模式（持续输入输出），同时保留单次计算模式向后兼容"

**关键原则**：**当前状态 → 目标状态 + 是否保留旧功能**。明确说明改造方向和兼容性要求，避免 AI 只做新功能而丢弃旧功能（虽然这次 AI 确实做了兼容，但明确要求更可靠）。

这类简短指令对环境探查任务足够，AI 会做全面检测。无需优化。

## 四、AI 侧反思

### 需改进之处

1. **技术选型应先问后做**：当用户提到技术名称（Qt/Vue/React 等），AI 应先确认具体实现方式而非直接选择最简方案。一句话确认可以避免一整轮的返工。

2. **应主动检查路径提示**：用户给出 `D:\QT` 路径时，AI 应第一时间 `ls` 该目录，从目录结构判断是 Qt SDK 还是其他安装，而非假设是 Python 绑定。

3. **应追问最终目的**：当用户要求"创建项目"时，AI 应主动问"最终部署目标是什么？"——网页部署、APK 打包、本地运行等目标会直接影响项目结构选择。

4. **跨版本修改应先列清单**：收到"全部修改"指令时，AI 应先列出计划新增的功能清单让用户确认，而非自行决定后直接执行。

### 做得好的地方

1. **向后兼容设计**：REPL 改造时保留了单次计算模式
2. **全面环境检测**：Qt 环境检查覆盖了多个维度
3. **关键配置识别**：Vue 项目中 `base: './'` 为 APK 场景做了正确配置
4. **根因分析**：按钮不可见问题准确识别为缺少显式 color 属性
5. **三版同步升级**：跨 Python/C++/JS 三版统一修改，逻辑一致性保持良好
